# ADR 0009: CityJSON Benchmark Corpus Design

## Status

Draft.

## Historical Note

This ADR predates the current publishable core split.
References to older crate names and broader transport plans should be read as
historical context, not as the current release surface.

## Context

We want a foundational CityJSON benchmark corpus that does three things at once:

1. exercises the CityJSON specification surface as completely as possible,
2. provides realistic shapes for correctness testing of common 3D city model operations, and
3. exposes the system-level costs that matter in practice: I/O, allocation, memory use, and de/serialization efficiency.

The existing building blocks already point in the right direction:

- `cjfake` can generate manifest-driven synthetic data.
- `serde_cityjson` already uses a manifest-backed synthetic benchmark catalog.
- `3DBAG` is a strong real-geometry base for geometry-sensitive operations.
- `cjindex` already prepares and reshapes 3DBAG-derived benchmark data into multiple storage layouts.

The missing piece is a coherent corpus design that keeps these concerns separate instead of collapsing them into one large, ambiguous dataset.

## Decision

The benchmark corpus should be treated as its own shared project, with its own repository, catalog, and derived artifacts.

The repository should define the corpus in three source kinds:

- `synthetic-controlled`
  Generated by `cjfake` from manifests. Used to isolate spec coverage, allocation behavior, and serialization shape.
- `real-geometry`
  Untouched `3DBAG` slices. Used where geometry correctness and real-world object structure matter.
- `real-geometry-enriched`
  `3DBAG` geometry augmented with synthetic attributes, metadata, appearance, or other non-geometric surfaces. Used to expand spec coverage without losing realistic geometry.

The benchmark taxonomy should be organized by bottleneck, not by CityJSON feature category:

- I/O-bound cases
- allocation-bound cases
- memory-efficiency cases
- de/serialization cases

Spec coverage remains important, but it is a control variable, not the primary benchmark axis.

## Corpus Shape

The corpus should have four layers:

### 1. Spec atoms

Small, focused cases that each exercise one spec surface or one narrow combination of surfaces.

These are the diagnostic cases. They should answer questions like:

- does this reader handle `transform` correctly?
- does this writer preserve appearance data?
- does this pipeline preserve hierarchy and multiple geometries per object?
- does stream packaging or extension handling still work?

### 2. Spec complete omnibus

A single deterministic fake dataset that covers the complete CityJSON surface as far as the generator and post-processing pipeline allow.

The existing `serde_cityjson` test data `cityjson_fake_complete` is a good
starting omnibus dataset for this role.

This dataset is for correctness gates, not performance ranking.

Use it for:

- parse correctness
- validation correctness
- roundtrip stability
- full-surface regression checks

This should not be the main performance benchmark because it is too mixed to diagnose a specific bottleneck.

### 3. Operation kernels

Medium-sized cases designed around common 3D city model operations:

- bounding box and extent queries
- object filtering and traversal
- hierarchy navigation
- feature extraction and merge/split workflows
- vertex compaction and reindexing
- semantic surface queries
- layout conversion

These should be realistic enough to exercise code paths, but still controlled enough to attribute cost.

### 4. Stress workloads

Large cases for throughput and memory pressure.

These should include:

- large `3DBAG` scans
- `feature-files` and other layout variants
- dense synthetic attribute or relation cases
- repeated-instance/template-heavy cases
- appearance-heavy serialization workloads

## Systems Programming Principles

The corpus should surface the costs that matter to systems work:

- I/O behavior and storage-layout sensitivity
- allocation count and allocation shape
- peak and steady-state memory use
- parse and write throughput
- roundtrip stability under realistic object graphs

That means each benchmark case should have one primary bottleneck and only a few secondary ones.

If a case changes geometry depth, attribute depth, layout, hierarchy, and appearance all at once, it stops being diagnostic.

## Role Split Between Projects

The repo boundary should be strict:

- `cjfake`
  Own synthetic generation and manifest ingestion for synthetic cases.
- `cjindex`
  Own real-data reshaping and layout derivation where that helps with benchmark preparation.
- benchmark corpus repo
  Own the canonical catalog, source manifests, derived corpus definitions, invariants, and released artifacts.

`cjfake` should ingest manifests directly for both the library and CLI paths, but it should not become the owner of the full derived-corpus pipeline.

## Manifest Model

The catalog should describe each case with fields that make the system cost explicit:

- source kind: `synthetic-controlled`, `real-geometry`, or `real-geometry-enriched`
- primary cost: `io`, `allocation`, `memory`, `deserialize`, or `serialize`
- secondary costs
- geometry validity: `dummy` or `real-preserved`
- representation: `cityjson`, `cityjsonfeature`, `jsonl`, `feature-files`, or another layout variant
- working-set scale
- supported operations
- correctness assertions

This keeps the corpus readable and makes it easier to reuse the same data for multiple tools.

## Consequences

Positive:

- the benchmark set has a clear engineering purpose
- correctness and performance concerns are separated
- real geometry stays available where it matters
- synthetic generation stays useful for controlled stress cases
- the corpus can be versioned and shared across projects

Tradeoffs:

- the benchmark pipeline will need its own repository and release process
- derived corpora will need clear ownership boundaries
- some cases will be generated, others derived, and a few may be hand-authored

## Current Conclusion

The working design so far is:

1. keep one complete-spec fake dataset as a correctness fixture,
2. keep smaller spec atoms for diagnosis,
3. use `3DBAG` as the geometry foundation for real-geometry workloads,
4. enrich `3DBAG` when broader spec coverage is needed,
5. organize the corpus by system bottleneck, and
6. host the benchmark corpus in its own repository.

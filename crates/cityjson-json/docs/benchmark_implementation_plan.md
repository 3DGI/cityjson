# Benchmark Implementation Plan

## Goal

Replace the current mixed benchmark setup with a cleaner benchmark suite that:

- measures only the current `serde_cityjson` crate
- covers both read and write paths
- uses deterministic synthetic fixtures generated by `cjfake`
- includes short case descriptions and throughput numbers in the README
- follows the general style of the `serde_json` benchmark suite: stable fixtures, setup outside the timed region, and throughput-based reporting

## Current State

The current benchmark surface still mixes multiple concerns:

- read benchmarks still compare against `serde_json::Value`
- legacy `serde_cityjson_legacy` references are still present in benchmark plumbing
- synthetic cases exist only as specs, not as generated fixtures
- write benchmarks are not yet first-class
- benchmark output is not yet rendered into README-ready tables

## Target Shape

The benchmark suite should be organized around two axes:

### Operation

- `read`
- `write`

### Dataset Family

- real-world regression cases
- synthetic architecture cases generated by `cjfake`

The real-world cases should remain available for regression tracking, but the synthetic cases should be the primary tool for explaining performance behavior.

## Benchmark Cases

### Read

Keep the current production datasets:

- `3DBAG`
- `3D Basisvoorziening`

Add synthetic cases that isolate specific deserialization cost centers:

- `geometry_flattening_best_case`
- `vertex_transform_stress`
- `attribute_tree_worst_case`
- `relation_graph_worst_case`
- `deep_boundary_stress`
- `composite_value_favorable_worst_case`

### Write

Use a subset that is likely to expose serializer behavior rather than parser behavior:

- `geometry_flattening_best_case`
- `attribute_tree_worst_case`
- `composite_value_favorable_worst_case`
- `appearance_and_validation_stress`

The write-specific stress case should exercise material/texture/template-heavy output and any validation work in the serializer path.

## `cjfake` Fixture Strategy

`cjfake` should generate deterministic fixtures from committed profiles. The benchmark repository should own the profiles, not the ad hoc output.

Each synthetic case should have:

- a stable case id
- a short description
- a seed
- an exact `cjfake` configuration
- an intended hot path or invariant

Suggested layout:

- `tests/data/generated/manifest.json`
- `tests/data/generated/README.md`
- `tests/data/generated/profiles/*.json`
- `tests/data/generated/cases/*.city.json`

If generated files are too large to commit, keep the profiles committed and generate the fixtures through `just` before benchmarking.

## Benchmark Harness

Split the benchmark harness by operation:

- `benches/read.rs`
- `benches/write.rs`

The harness should:

- prepare input outside the timed closure
- use `Criterion` throughput reporting for every benchmark group
- benchmark `serde_cityjson` against `serde_json::Value` on the read side
- benchmark `serde_cityjson` against `serde_json::to_string` on the write side
- avoid measuring fixture generation
- avoid mixing legacy comparison groups into the current suite

### Read Benchmarks

The read benchmarks should measure:

- `serde_cityjson::from_str_owned`
- `serde_cityjson::from_str_borrowed` where valid
- `serde_json::Value`

Throughput should be based on input bytes.

### Write Benchmarks

The write benchmarks should measure:

- `serde_cityjson::to_string`
- `serde_cityjson::to_string_validated`
- `serde_json::to_string`

Throughput should be based on output bytes.

## Reporting

Add reporting that produces both plots and README-ready Markdown tables.

The report should include:

- case name
- brief description
- input or output size
- mean time
- throughput
- relative factor compared with `serde_json::Value`

The report should be deterministic enough that it can be regenerated and pasted back into the README.

## README

The README should eventually include:

- a short benchmark methodology section
- a compact read benchmark table
- a compact write benchmark table
- a one-line description for each synthetic case
- throughput numbers alongside raw timings

The README should be the consumer-facing summary, while the detailed case definitions stay in the manifest.

## Rollout Plan

### Phase 1: Remove Legacy Benchmark Plumbing

- remove `serde_cityjson_legacy` benchmark dependencies
- delete legacy benchmark groups
- remove legacy download and plot assumptions

### Phase 2: Make Synthetic Cases Real

- freeze the `cjfake` profiles
- generate the fixtures deterministically
- validate that the cases match the intended cost centers

### Phase 3: Split Read and Write Benchmarks

- create separate benchmark entrypoints
- establish throughput reporting
- keep setup outside the timed region

### Phase 4: Add Reporting

- produce Markdown tables
- produce plots
- wire report output into the README workflow

### Phase 5: Publish Results

- run the full benchmark suite
- curate the README summary
- keep the plots and case descriptions in sync with the manifest

## Acceptance Criteria

The implementation is done when:

- the benchmark suite no longer depends on `serde_cityjson_legacy`
- read and write benchmarks are separate and deterministic
- synthetic cases are generated from `cjfake` profiles
- benchmark output includes throughput numbers
- the README contains short case descriptions and measured results
- the benchmark workflow can be regenerated without manual editing of the measured data

## Risks

- Synthetic cases may become too tied to current implementation details if they are not designed around stable cost centers.
- Write benchmarks may need a different set of cases than read benchmarks.
- Large generated fixtures may be expensive to keep in the repository, so the generation workflow must stay simple.

## Notes

The benchmark design should stay focused on the current architecture. The point is not to preserve historical benchmark behavior, but to make performance regressions and improvements easy to interpret.
